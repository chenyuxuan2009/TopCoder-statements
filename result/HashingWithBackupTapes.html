Problem Statement for "HashingWithBackupTapes"

Problem Statement

We have a hash table with H buckets, numbered 0 to H-1. Each bucket contains a linked list of elements stored there. Initially, each bucket is empty.

When adding an element to a bucket that already contains x elements, the element is compared to all other elements in the bucket and if it's a new element, it is then appended to the list. The worst case is clearly adding a new element, which takes x+1 steps and increments the size of the bucket.

When analyzing the time complexity we will be interested in the worst case. Hence, you may assume that whenever an element is added into a bucket, that element is always a new element.

So, for example, inserting four elements into the same (initially empty) bucket takes a total of 1+2+3+4 = 10 steps and leaves us with a bucket that now contains four distinct elements.


We have T empty backup tapes available.

At any moment you can take the content of any single bucket in the hash table, store its content on an empty tape, and empty the bucket.

Tape backup can be done in the background and emptying the bucket can be done in constant time, so we'll assume this operation takes no time at all.


Doing a backup is good because it will speed up the processing of future elements that get stored in the same bucket.

On the other hand, doing a backup is bad because the new elements that are later added to the bucket won't get compared to the ones that got backed up earlier.

Thus, at the very end of our process (i.e., after the very last insertion into the table) we need to fix this issue we created by backing up. We will bring back all the backup tapes we used. For each bucket that got backed up at least once, we now need to take all the elements that ever were in that bucket (i.e., all backups of that bucket + the elements that are in that bucket at the end) and we have to sort them to get rid of possible duplicates. If there is a total of y &gt; 0 elements in the bucket's history, this process takes 2*y*ceiling(log2(y)) steps.


We know that we are going to process a sequence of insertions. The sequence consists of N phases, numbered 0 to N-1. In phase i we are doing cnt[i] individual insertions, one after another, all of them into the same bucket: bkt[i].

We want to optimize the worst-case number of steps needed to process these insertions by potentially using some backups. If we use some backups, the total number of steps includes the final sorting of each bucket that was backed up.

Calculate and return the optimal (i.e., smallest) worst-case number of steps in which we can process the given sequence of insertions.


In order to keep the input size small, the input is pseudorandom. Please use the pseudocode below to generate the arrays cnt[] and bkt[]: 


who = an empty array
for i = 0 .. H-1:
    repeat W[i] times:
        who.append(i)

state = seed

for i = 0 .. N-1:
    state = (state * 1103515245 + 12345) modulo 2^31
    bkt[i] = who[ state modulo length(who) ]
    state = (state * 1103515245 + 12345) modulo 2^31
    cnt[i] = 1 + (state modulo L)

Notes

  • Remember that the number of available backup tapes limits the number of times you can back a bucket up.
  • Only the buckets that got backed up one or more times need to be sorted at the end.
  • Sorting a bucket that never contained any elements takes zero steps.
  • The reference solution does not depend on the input being pseudorandom.
  • The function ceiling(x) returns the smallest integer that is at least x. For example, ceiling(3.1) = ceiling(3.9) = ceiling(4) = 4.

Constraints

  • H will be between 1 and 100, inclusive.
  • T will be between 0 and 300, inclusive.
  • N will be between 1 and 100,000, inclusive.
  • L will be between 1 and 1000, inclusive.
  • W will be have exactly H elements.
  • Each element of W will be between 1 and 100, inclusive.
  • seed will be between 0 and 2^31 - 1, inclusive.
